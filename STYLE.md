# alswl 博客写作风格指南

本文档总结 alswl 的博客写作风格，供 AI 辅助写作时参考。

## 作者画像

- 笔名 alswl，互联网集团工程师，从事容器调度方向
- 博客运营自 2007 年至今，累计 456 篇文章
- 主要写作领域：软件工程（coding）、效率工具（efficiency）、思考随笔（thinking）、观点输出（viewpoint）、团队管理（management）、年终总结（life）
- 早期高产（2009 年 197 篇），近年转为低频深度长文（每年 3-8 篇）

## 语言与用词

- **中文为主**，技术术语保留英文原文，如 DevOps、SRE、GTD、Sprint、HTTPS
- 中英文混排自然，不刻意翻译已有共识的技术名词
- 术语首次出现时给出英文全称和中文解释，如「SRE 全称是 Site Reliability Engineering」
- 口语化但不随意，偶尔使用网络用语增加亲切感，如「给了我一个巨大的逼兜」「拉了一泡稀的」「尸位素餐」
- 善用四字短语和对仗句式，如「对预期内问题建立预案，对计划外变更保持弹性」「热烈又恬静，深刻又朴素」
- 偶尔自嘲和幽默，如「前端实习生 // 社区打补丁专家 // Kubernetes 清洁工 // YAML 资深专家」

## 文章结构

### 标题

- 技术文标题简洁直接，常用「谈」「浅谈」「也谈」开头，或直接命名主题
- 喜欢用副标题或英文补充，如「架构设计 the Easy Way」「从 Toodledo 到 Obsidian Tasks - 我的 GTD 最佳实践」
- 标题常带有观点倾向或实用定位，如「实用软件项目管理最佳实践」「如何逃离死海效应」

### Frontmatter 格式

```yaml
---
title: "文章标题"
author: "alswl"
slug: "url-slug"
date: YYYY-MM-DDTHH:MM:SS+08:00
categories: ["coding"]
tags: ["tag1", "tag2"]
---
```

- categories 单选，从 coding / life / efficiency / thinking / viewpoint / managment / fun 中选
- tags 使用小写英文为主，偶尔中文标签如「读书笔记」
- slug 使用英文短横线连接

### 正文结构模式

技术文章通常遵循以下结构：

1. **开篇引入**：从个人经历、近期事件或常见误区切入，建立读者共鸣
2. **概述/定义**：用一段话概括全文要讲什么，明确范围和边界（「本文仅针对...不涉及...」）
3. **问题驱动**：先讲遇到什么问题，再讲如何解决（「问题不可怕，定义清楚问题就成功了一半」）
4. **分层展开**：用 `##` 二级标题分大章节，`###` 三级标题分小节，层次清晰
5. **案例穿插**：每个观点配一个实际案例或范例，来自真实工作经验或知名开源项目（Prometheus、Kubernetes、AWS）
6. **小结收束**：每个大章节末尾有小结，提炼要点
7. **番外/扩展阅读**：文末附相关书单、参考链接、技能点清单等延伸内容

生活/年终总结类文章结构：

1. 以照片或感性文字开篇
2. 按维度分块：生活、工作、业余、读书
3. 读书部分会附豆瓣链接和封面图，配 2-3 句精炼点评
4. 以一句期望或感悟收尾

## 写作手法

### 问题驱动叙事

几乎所有技术文章都从「问题」出发，而非从「技术」出发。先让读者感知痛点，再展开解决方案。

### 实用主义（Pragmatic）基调

- 强调「可操作性」和「立即可上手」，反对过度理论化
- 常出现「80/20 原则」的应用——关注核心 20% 场景
- 会主动声明局限性：「没有银弹」「不一定适合所有场景」「免责申明」
- 推崇《The Pragmatic Programmer》理念

### 对比与类比

- 善用对比澄清概念，如 DevOps vs SRE、Push vs Pull、流程图 vs 时序图
- 用通俗类比解释抽象概念，如「将大象装进冰箱」解释架构设计步骤、「死海效应」解释人才流失

### 个人经验为锚点

- 大量引用自身工作经验作为论据，而非纯粹引经据典
- 坦诚分享失败和困难，如「我甚至感到自己一度快要被压垮了」「几次因为项目无法交付失眠」
- 推荐工具时会说明自己使用了多久、付费了多久、为什么迁移

### 口诀/公式化总结

- 喜欢将方法论凝练为口诀或公式，便于记忆：
  - 「架构看问题、需求看用例、设计看模型、细节看时序」（20 字口诀）
  - 「工作量 =（最乐观 + 最悲观）/ 2」
  - 「速度 = 总量 / 时间」
  - 「写下来」（三个字的核心武器）

## 排版与格式

### Markdown 用法

- 使用 `<mark>` 标签高亮关键句和核心观点
- 使用 `<small>` 标签标注图片来源
- 粗体 `**` 用于强调关键词和重点概念
- 引用块 `>` 用于定义、引文、模板示例
- 表格用于对比、分类、清单展示
- 代码块用于展示配置、查询语法、技术实现
- 列表（有序/无序）大量使用，层级清晰
- `<!-- more -->` 用于标记摘要截断点（部分文章）

### 图片使用

- 文章头图通常来自 Pixabay 等免费图库，或使用产品截图
- 技术文章大量使用架构图、时序图、截图辅助说明
- 图片路径格式：`../../static/images/YYYYMM/filename.png`
- 图片下方用 `<small>` 标注来源

### 链接引用

- 内链引用自己之前的博客文章，形成知识网络
- 外链引用维基百科、官方文档、豆瓣读书、GitHub 仓库
- 参考资料集中放在文末 `## Ref` 或 `## 扩展阅读` 章节

## 情感基调

- **克制但不冷淡**：技术文章以理性分析为主，但会穿插个人感受
- **坦诚直率**：敢于表达强烈观点，如「我讨厌开会，甚至从某种意义上我痛恨开会」
- **温和的反骨**：质疑主流但不对抗，如对运营商劫持的吐槽、对 SaaS 可靠性的质疑
- **感性收尾**：技术文最后一段常回归人文关怀和个人感悟

## 写作禁忌（观察所得）

- 不写纯搬运/翻译类文章，一定有自己的理解和实践
- 不写没有实操经验支撑的纯理论文章
- 不使用过多 emoji（仅在年终总结的 side project 介绍中偶尔使用）
- 不写引战或纯吐槽内容，即使有批评也基于事实和分析
- 不堆砌术语炫技，出现专业概念时一定解释清楚

## 写作辅助提示

当为 alswl 代笔或辅助写作时，请注意：

1. **从问题切入**，不要上来就讲方案
2. **附带真实案例**，优先使用开源项目或行业实践作为例证
3. **保持实用基调**，声明适用范围和局限性
4. **中英文混排**，技术术语不强行翻译
5. **用 `<mark>` 高亮核心观点**，用粗体强调关键词
6. **每个大章节配小结**，文末有扩展阅读
7. **语气克制但有态度**，可以有幽默感但不浮夸
8. **文件命名**：`YYYY-MM-DD-slug.md`，slug 使用英文短横线
